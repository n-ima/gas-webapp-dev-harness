# 使い方ガイド（実例つき）

[README.md](README.md) が全体構造の説明だとすると、このドキュメントは
「実際に手を動かすと何が起きるか」を、具体的な例で最初から最後まで追ったものです。
これから使う人は、まずこの一本の流れを眺めてから始めてください。

## 登場人物と前提

- 以下のいずれかのエージェント環境（このガイドはCopilotの画面を例に書いているが、
  コマンド名・フェーズの流れは3環境とも同じ）:
  - VS Code + GitHub Copilot Chat（Custom Agents / Agent Skills 対応バージョン）
  - Claude Code（`/00-start-project` 等が同名のスラッシュコマンドとして使える。
    ハンドオフボタンは無いので、案内されたコマンドを新しいセッションで実行する）
  - Antigravity（同名のワークフローとして使える。フックによる機械的ガードは
    効かないため、破壊的操作の確認は指示レベルで担保される）
- 題材例: 「社内向けの休暇申請ツールを作りたい」という小さな社内アプリ
  （実際のプロジェクトでは、あなたのアプリの内容に読み替えてください）

## 0. 準備：このテンプレートを入手する（3経路）

1. **GitHubテンプレート（推奨）**: リポジトリの「Use this template」で新規リポジトリを作成し、クローンする。
2. **ZIP配布**: 受け取ったZIPを展開後、次の順で初期化する。
   `git init -b main` → 最初のコミット → `git remote add origin <案件リポジトリURL>` → push。
   **配布する側への注意**: 作業フォルダをそのまま圧縮しない（ハーネス本体を指す `.git` が
   混入し、受領者が誤って本体へpushする事故が起きうる）。安全なZIPは
   `git archive --format=zip -o ../<name>.zip HEAD` で作る（`.git` や `.venv` 等が混入しない）。
3. **既存リポジトリへの導入**: `.github/`（agents/prompts/skills/hooks/instructions）、
   `AGENTS.md`、`docs/`、`requirements/`、`.gitattributes` をコピーする。

いずれもVS Code でフォルダを開けば、`.github/agents`, `.github/prompts`, `.github/skills`,
`.github/hooks` は既定で自動検出される（`.vscode/settings.json` は
共有ロケーションを増やしたい場合のみ編集する）。

## 1. 要件メモを書く（人がやること：ざっくり書くだけ）

[requirements/memo.md](requirements/memo.md) を開き、思いつくままに書きます。丁寧である
必要はありません。

```markdown
休暇申請を今Excelでやってて、承認の抜け漏れが多い。
社員が申請 → 上長が承認 → 総務が確認、くらいの流れがWeb上でできればいい。
社員は50人くらい。社内ネットワークからしか使わない想定。
```

これだけで十分です。ここから先はエージェントが質問して埋めていきます。

## 2. `/00-start-project` で現状確認する

Copilot Chat を開き、入力欄に `/00-start-project` と入力して実行します。
このプロンプトは frontmatter で `agent: orchestrator` に固定されているため、
今どのエージェントを選択していても自動的に `orchestrator` として動きます。

初回はこう返ってくるはずです。

- `docs/00-overview/progress.md` が無いのでその場で作成（機械的な作業なので確認なし）
- 要件定義〜リリースの状態表（すべて「未着手」）
- 推奨アクション: 「要件定義を進める」ボタン（`requirements` エージェントへのハンドオフ）

## 3. 要件定義（人がやること：質問に答える）

「要件定義を進める」ボタンを押すと、`requirements` エージェントに切り替わり、
プロンプトが入力欄に入った状態になります（`send: false` なので、内容を確認してから
自分でEnterを押します）。

`requirements` エージェントは `.github/skills/requirements-elicitation/SKILL.md` の
チェックリストに沿って、カテゴリごとにまとめて質問してきます。例えば：

> - **目的・背景**: Excel運用のどこが具体的に困っていますか？（承認漏れの頻度は？）
> - **利用者**: 社員・上長・総務、それぞれの役割と人数は？
> - **機能要件**: 申請フォームの項目、承認フローの段階数、通知は必要ですか？
> - **非機能要件**: 社内ネットワーク限定とのことですが、認証方式は？既存の社内認証基盤はありますか？
> - **デプロイ環境（重要）**: どこで動かしますか？社内サーバー？クラウド？CI/CDはありますか？
>   認証情報は誰が管理していますか？

**ここが最も人の負荷が高いフェーズです。** 特に「デプロイ環境」の質問には具体的に答えてください。
ここが曖昧だと、後述するリリースフェーズの自動実行が成立しません。

回答すると、以下が `docs/01-requirements/` に作成されます。

- `requirements.md`（ユーザーストーリー + 受け入れ条件）
- `nfr.md`（非機能要件）
- `glossary.md`（用語集）
- `environment.md`（デプロイ環境・自動化してよい範囲。**最重要**）

最後に「この内容で要件定義完了としてよいですか？」と聞かれるので、内容を確認して承認します
（不足があればここで指摘する）。

## 4. 設計（人がやること：技術選定の相談に答える＋最終承認1回）

要件定義の承認後、エージェントが「**新しいチャットで `/03-design-architecture` を
実行してください**」と案内します。その通りに新しいチャットを開いてプロンプトを実行すれば、
自動的に `design` エージェントとして起動します（手動でエージェントを切り替える必要は
ありません）。会話が短い小規模案件なら「設計フェーズへ進む（このチャットで続行）」
ボタンで続けることもできます。

`design` エージェントはまず、要件が十分に詳細か自分でチェックします
（受け入れ条件・NFR・environment.mdが埋まっているか）。**もし薄ければ、ここで
要件定義エージェントに差し戻されます。** 十分であれば設計を進めます。

技術スタックのような「後戻りが難しくトレードオフが利用者の価値観次第」な選択だけ、
2〜3案を比較表で提示してきます。

> 案A: Node.js + Express + SQLite（小規模・低コスト、社内サーバーに簡単に置ける）
> 案B: Next.js + PostgreSQL（将来の拡張性は高いが、今回の規模には過剰気味）
> どちらにしますか？

それ以外の設計判断（コンポーネント分割、API設計、詳細設計）は自分で決めて完成させます。

**画面があるアプリの場合は、ここで画面モックアップの視覚確認が入ります。**
`docs/02-design/ui/` に主要画面の自己完結型HTML（ダブルクリックでブラウザ表示可能）が
作られるので、実際に開いて見た目を確認してください。「ヘッダーを固定して」「この色は
濃すぎる」のようにチャットで指示すれば修正されますし、HTMLを直接編集して返しても
構いません（デザインに慣れている場合はこちらが速い）。閲覧に トークンは掛からないので、
納得いくまで見直せます。合意したモックアップが画面の「正」となり、実装はこれに従います。

一区切りついたら、**最終確認は1回だけ**：「この設計で実装フェーズに進んでよいですか？」。

ここで承認すると、`docs/02-design/architecture.md`・ADR・詳細設計（＋UIモックアップ）が
揃った状態になります。

## 5. 実装〜テスト（人がやること：基本ノータッチ）

設計の承認後、案内に従って**新しいチャットで `/05-implementation-plan` を実行**すると、
あとは自動で進みます（設計チャットが短ければ「実装フェーズへ進む（このチャットで続行）」
ボタンでも同じ動作です）。

1. `implement` エージェントが `docs/03-implementation/tasks.md` にタスクを分解する
   （ここだけ一度、粒度・順序の確認が入ります）。
2. 承認すると、先頭のタスクから**確認なしで**実装が進みます。ただし `implement` 自身は
   コードを書きません。タスクを1つ実装するたびに `task-worker` サブエージェントを
   `runSubagent` で呼び出し、まっさらな独立コンテキストで実装させます
   （長い会話ほどモデルの精度が落ちる"context rot"という現象への対策です。
   `implement` 自身は「次はどのタスクか」を判断するだけの軽いコーディネーターに留まります）。
3. 全タスク完了で `test` エージェントに自動で引き継がれます（ボタンを押す必要なし）。
4. `test` エージェントがテスト計画を作り、実行し、失敗があれば自動で `implement` と
   往復して直します。
5. 全テストが通ったら、**リリース前に必ず** `reviewer` サブエージェントが独立コンテキストで
   1回呼ばれます（実装した本人がそのまま自己承認しないようにするためです）。
   発見事項は `docs/04-test/security-review-report.md` と `test-report.md` に記録されます。
6. 問題なければそのまま `release` エージェントに自動で引き継がれます。

**この区間で人に確認が来るのは、次の場合だけです。**

- 設計・要件だけでは判断できない真のブロッカーに当たったとき
- `.github/hooks/` が `git push`・タグ付け・force系操作・`rm -rf` 等を検知して確認を求めるとき
  （実装フェーズでこれらを行うことは通常ないはずですが、保険として効きます）
- `reviewer` が CRITICAL/HIGH の問題を報告し、実装との往復で2回解消しなかったとき

## 6. リリース（人がやること：人手必須の作業だけ）

`release` エージェントは `docs/01-requirements/environment.md` を見て、
「今回は社内サーバーへのデプロイで、SSH経由でのデプロイスクリプト実行は自動化してよい、
ただし社内DNSへの登録申請は総務経由の人手作業」といった分類に従って動きます。

- `.github/skills/deploy-<environment>/SKILL.md` がまだ無ければ、その場で作成しながら
  進めます（次回以降の同種プロジェクトで自動化されるように）。
- 自動化してよい作業（ビルド、パッケージング、デプロイスクリプト実行等）はそのまま実行されます。
- 人手必須の作業（DNS登録申請など）に来たら、そこで止まり「総務にこの申請をしてください」と
  具体的に提示してきます。完了を伝えると続きを進めます。
- `git push` / タグ付けは `.github/hooks/` により必ず一度確認が入ります
  （environment.mdの分類に関わらず）。

最後に全フェーズ完了として `docs/00-overview/progress.md` を更新してよいか聞かれます。

## 7. 振り返り（人がやること：結果を読んで、ハーネスに還流する）

リリース完了時に `/10-retrospective` の実施を提案されます。実行すると、開発中の
摩擦（エージェントが詰まった箇所、訂正が必要だった箇所、フックの誤検知など）が
「ハーネス改善 / プロジェクト固有 / 一過性」に分類され、ハーネス改善分は
具体的な改善提案表として `docs/06-retrospective/retrospective.md` にまとまります。

この改善提案と、開発中に作られた再利用可能なSkill（`deploy-vercel`等）を
ハーネスのテンプレートリポジトリに持ち帰って適用すれば、**次のプロジェクトは
今回の学びが反映された状態から始まります**。これがこのハーネスの成長ループです。

なお開発中も、エージェントに同じ趣旨の訂正を2回した場面などでは
`docs/00-overview/learnings.md` に教訓が自動的に記録され、以後のセッションに
自動注入されます。「前に言ったのにまた同じ間違いをする」と感じたら、
learnings.mdに記録されているか確認してください（直接追記しても構いません）。

## まとめ：人が実際にやること

| フェーズ | 人の作業 | 目安の手間 |
|---|---|---|
| 準備 | メモを雑に書く | 数分 |
| 要件定義 | 質問にきちんと答える（特に環境情報） | 一番かかる。ここで手を抜かない |
| 設計 | 技術選定の相談に答える＋最終承認1回 | 数分〜数十分 |
| 実装〜テスト | 基本何もしない（ブロッカー時のみ対応） | ほぼゼロ |
| リリース | 人手必須の作業＋push/tagの最終確認 | 環境次第 |

## Q. 要件定義書や設計書を後から自分で修正しても大丈夫？

**大丈夫です。** 各エージェントは会話の記憶ではなく `docs/` のファイルを毎回読み直すので、
`requirements.md` や `architecture.md` を直接エディタで開いて直しても、次にエージェントが
動いたときはその内容を前提に動きます。会話を続ける必要も、最初からやり直す必要もありません。

ただし2点だけ注意してください。

- **軽微な修正（数値・表記の訂正など）は手動でOK**ですが、**スコープや決定内容が変わる
  ような修正は、エージェントに「ここをこう変えたい」と伝えて直してもらう方が安全**です。
  エージェントは変更が波及する範囲（非機能要件・環境情報・ADR・下流フェーズの成果物）まで
  一緒に確認・更新してくれますが、手動編集だとそこが漏れます。
- すでに「完了(done)」として承認済みの文書を編集すると、`.github/hooks/` の
  `warn-stale-gate` フックが「後続フェーズとの整合を確認してください」という注意を
  自動で出します。これが出たら、該当する後続フェーズのエージェントに一声かけて
  整合を確認してもらってください。

## 困ったときに確認すること（Preview機能ゆえの注意）

このハーネスが使っている Agent Hooks / Agent Plugins は執筆時点でVS CodeのPreview機能です。

- ハンドオフボタンが出ない・Hooksが発火しない → 使っているVS Code/Copilot拡張のバージョンが
  対応しているか、組織ポリシーで無効化されていないか確認してください
  （[.github/hooks/README.md](.github/hooks/README.md) 参照）。
- Hooksが期待通り動かない → `.github/hooks/scripts/` のペイロード解析部分は
  一次情報から確認した範囲の実装です。実際のペイロード形状に合わせて調整してください。
- それでも `AGENTS.md` に書かれた指示レベルのルールは、Hooksが効かない環境でも
  エージェント自身の判断で守られるように書いてあります（二重の安全網）。

## コストを意識した使い方

- 実装〜テストのように呼び出し回数が多いフェーズは `model: auto` にしてあるので、
  基本的には何もしなくてもコスト効率の良いモデルが自動選択されます。
- 設計フェーズやレビューで「もう少し賢いモデルに切り替えたい」と感じたら、
  そのときだけ手動でモデルを切り替えてください（常時強いモデルを使う必要はありません）。
- Copilot ChatでSubagent（`reviewer`, `task-worker`）を呼んだ際は、その部分にカーソルを
  合わせると使用したAI Creditが確認できます。想定より高い場合は、`reviewer`を呼ぶ頻度
  （test.agent.mdの方針: 1テスト完了につき原則1回）を見直してください。

## セッション分割表（いつ新しいチャットにするか）

同一セッションの継続は毎ターン会話履歴全体を再送するため、履歴が長いほど1ターンの
コストが増え、想起精度も落ちます（"context rot"）。`docs/` が正の状態なので、
セッションを切っても失うものはありません。

| 作業 | セッション |
|---|---|
| `/00`, `/98`, `/99` 進捗確認・ヘルプ | どこでもよい |
| `/01`-`/02` 要件定義 | 新規 |
| `/03`-`/04` 設計 | 新規（要件と別） |
| `/05` 実装計画 → `/06` 実装 → `/07`-`/08` テスト | 1セッションでよい（コーディネーターは軽量、実装はtask-workerに隔離） |
| `/09` リリース | 自動継続（ノンストップ設計）。手動で再開する場合は新チャット+`/09`でも同じ動作 |
| `/10` 振り返り | 新規 |
| リリース後の改修サイクル開始 | 必ず新規 |

**継続中でも、次の4条件のいずれかで新チャットに切り替えてください**
（エージェント自身も該当時に切り替えを提案します）。

1. このセッションでの完了タスクが10個を超えた
2. 実装⇔テストの差し戻しが2往復を超えた
3. 応答の劣化（話が噛み合わない・同じ確認の繰り返し）を感じた
4. 中断して時間を空ける

再開方法は「新しいチャットで該当プロンプトを実行するだけ」です。
実装フェーズで `task-worker` を使っているのも、この対策を自動化した仕組みです。

## リリース後の機能改修・バグ修正

全フェーズをやり直す必要はありません。AGENTS.md「差分駆動の原則」の4分類に従います。

- **要件・設計に影響する変更**: 新しいチャットで該当フェーズのプロンプト
  （`/02` または `/03`）を実行し、変更部分だけ再ゲートする。影響範囲は
  トレーサビリティ表から自動的に辿られます。
- **設計変更を伴わないバグ修正**: 新しいチャットで `/06-implement-task` を実行し、
  再現テスト先行で最小修正。文書修正・再ゲートは不要です。
- **緊急ホットフィックス**: 直接修正を許容しますが、再現テスト・PR・人の承認は省略せず、
  該当文書に「実装乖離あり（解消期限）」を注記してください（期限内に文書を更新して整合回復）。
